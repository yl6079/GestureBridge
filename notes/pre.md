# GestureBridge Presentation Outline

## 一、做了什么：实现手语识别与双向交互系统

GestureBridge 是一个部署在 Raspberry Pi 等边缘设备上的离线辅助系统，实现 **手语 ↔ 语音 ↔ 文字** 的双向交互，同时支持教学训练场景。

### 系统整体目标：

- 帮助手语使用者与普通用户交流  
- 提供低成本、离线、隐私友好的解决方案  
- 支持课堂展示与学习实践  

---

## 二、系统整体架构

### 边缘双设备协同

### 1. 前端感知层（ESP32）

使用 ESP32 搭配轻量模型（Edge Impulse）进行粗略动作识别。

通过状态机判断是否需要唤醒主系统。

#### 作用：

- 降低树莓派持续运行负载  
- 节省功耗  
- 提升响应效率  

---

### 2. 主计算层（Raspberry Pi）

Raspberry Pi 负责：

- 摄像头实时图像采集  
- 手语识别推理  
- 本地语音识别  
- 语音播报  
- Web UI 展示  

---

## 三、核心功能模块（每个模块可配 Demo）

## 模式1：手语翻译模式（Sign to Speech）

用户通过摄像头做出 ASL 手势动作。

### 系统输出：

- 实时文字结果  
- 语音播报结果  

### 识别模型采用：

- MobileNetV3 TFLite 模型  
- 辅助 Landmark / 第二模型融合（提升准确率）

### 目标：

提高实际交流效率。

---

## 模式2：语音转文字与手语模式（Speech to Sign）

用户说话后，系统通过本地部署语音模型识别内容。

### 输出：

- 文字结果  
- 字母拼写 / 手语指示  

### 语音识别采用：

- Vosk 本地部署

### 目标：

帮助普通用户向手语使用者表达信息。

---

## 模式3：学习训练模式（Learning Practice）

复用手语识别能力，增加交互式学习流程：

- 系统给出目标字母  
- 用户做动作  
- 实时反馈正确 / 错误  

### 目标：

辅助初学者学习手语。

---

## 四、关键挑战与设计决策

## 1. 双设备协同控制复杂

### 问题：

ESP32 与树莓派之间需要可靠切换与唤醒机制。

### 决策：

引入状态机控制流程，统一管理待机、激活、运行状态。

### 结果：

系统更稳定，资源利用率更高。

---

## 2. 实时性 vs 准确率

### 问题：

INT8 模型推理更快，但识别稳定性一般。

测试后发现：

FP32 模型虽然稍慢，但准确率提升明显，额外开销可接受。

### 决策：

默认采用 FP32 模型部署。

---

## 3. 单模型鲁棒性不足

### 问题：

单一视觉模型在角度变化、遮挡、光线变化下效果有限。

### 决策：

在资源允许时，引入第二模型 / Landmark 融合进行结果补充。

### 结果：

提高复杂场景下识别准确率。

---

## 4. 云端 API vs 本地部署

### 问题：

云端语音 API 存在：

- 网络依赖  
- 延迟  
- 隐私问题  

### 决策：

改为本地部署 Vosk。

### 结果：

实现真正离线可用系统。

---

## 五、最终成果

GestureBridge 最终实现了：

- 边缘端实时手语识别  
- 双向语音交互  
- 学习辅助模式  
- 完全离线运行  
- 可在低成本设备部署  

展示了 AI 在无障碍沟通场景中的实际应用价值。
